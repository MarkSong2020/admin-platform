# P1.5 — RBAC 绑定管理 API + rbac_write 审计织入（decision-log）

> 2026-06-09。源自当日「多视角深度 review」(5 subagent + Codex high) 的两个 🔴：
> ① 绑定管理 API 缺失（RBAC 不可运营）② rbac_write 死事件（spec §13.3 要求但零 emit）。
> 设计经 **Claude × Codex medium PK 收敛**（无实质分歧）。本文件冻结接口/审计契约，实现后回填。

## 1. 背景缺口（review 实证）

- 绑定 repository 方法齐全（`set_user_roles`/`set_role_depts`/`set_role_menus`/`set_user_posts`，均带 advisory lock），但**无 service/API 出口、DTO 无 *_ids 字段** → 非超管权限只能 seed/脚本写，管理端不可配。
- `rbac_write` 已在 `audit/events.py` EventType 枚举 + `test_audit_event_contract.py` 参数化，但**无任何 emit 调用点** = dead event + dead test。spec `docs/specs/2026-06-05-p1.0-rbac-mechanism.md` §13.3 列其为 P1 必落三类事件之一。

## 2. 接口契约（子资源 PUT + 成对 GET）

**核心裁决：子资源 PUT 优于扩展 PATCH**。绑定是全量替换（`set_*` 先删后插），PATCH merge 语义会让"空数组=不改还是清空"靠 `model_fields_set` 维持、且标量改与绑定改混在一起使 404/422/审计/权限都变脆。对标若依，分配角色/菜单/部门本就是独立管理动作。

| 写 endpoint | body | 读 endpoint | 权限点（复用，不新增 assign） |
|---|---|---|---|
| `PUT /api/v1/users/{user_id}/roles` | `{role_ids:[…]}` | `GET …/roles` | 写 `system:user:edit` / 读 `system:user:query` |
| `PUT /api/v1/users/{user_id}/posts` | `{post_ids:[…]}` | `GET …/posts` | 写 `system:user:edit` / 读 `system:user:query` |
| `PUT /api/v1/roles/{role_id}/menus` | `{menu_ids:[…]}` | `GET …/menus` | 写 `system:role:edit` / 读 `system:role:query` |
| `PUT /api/v1/roles/{role_id}/depts` | `{dept_ids:[…]}` | `GET …/depts` | 写 `system:role:edit` / 读 `system:role:query` |

- 响应 `200 BindingRead`（回显**去重后**最终 ids，便于管理端确认）；空数组 = 清空（幂等）。
- 写入流程：① 校验主体 user/role 存在 → 404；② 绑定 ids 全量存在校验（去重后比数量），缺失 → 422 错误码 `{domain}.<X>_IDS_INVALID`；③ 调既有 `set_*`；④ emit `rbac_write`。
- `UserRead` **不**回显 roles/posts（避免列表 JOIN/N+1/权限面放大）；详情走子资源 GET。
- active 状态：**允许绑定 disabled 对象**（管理端可预配置），授权读取时 provider 仍只让 active 生效（已有行为）。

## 3. 审计契约（rbac_write，service 级 helper）

**覆盖范围 = 所有 RBAC 管理写**：user/role/menu/dept/post 的 create/update/delete + 四类绑定 PUT。理由（Codex 红线）：改 `role.data_scope`、禁用角色、删菜单、移部门都直接改变授权结果，只审计绑定会漏高风险写。

**形态**：`audit/emit.py` 新增 `emit_rbac_write(actor, action, target, *, status, http_status, error_code=None, metadata=None)` helper；各 service 写路径**显式调用**（不用装饰器——装饰器拿不到 domain target / 差异字段 / 失败码 / 回滚语义）。

| 字段 | 取值 |
|---|---|
| `event_type` | `rbac_write` |
| `action` | 权限点或细动作，如 `system:role:edit` / `system:role:bind_menus` |
| `actor` | `AuditActor(user_id=…)`，从 api 层 CurrentUser 传入 service（**不 import Request**） |
| `target` | `type=user\|role\|menu\|dept\|post\|binding`，`id`=主资源 id，`display`=name/code/username |
| `result` | 成功 `success+200/201/204`；业务失败 `failure+404/422/409+error_code` |
| `metadata` | 仅非敏感差异摘要（`after_ids/added/removed/counts`/字段名）；**禁** password/token/header |
| `risk_level` | `medium`（RBAC 写中敏感度） |

**失败也审计**：service helper 包 `try/except AppError` → emit `failure` 后 **re-raise**（不吞异常、不阻断原业务错误）。

## 4. 数据权限约束（写侧越权防护）

- `user-role` / `user-post`：目标 user 继续受操作者 `user.data_scope` 约束（非超管不能给不可见 user 改绑定）→ 404（不泄露存在性，与 dept 同口径）。
- `role-dept`：dept_ids 受操作者数据范围约束——非超管只能绑**可见**部门，否则 403 `auth.FORBIDDEN_BY_SCOPE`。
- `role`/`menu` 是全局配置资源（不套 data_scope，仅靠权限点管控），与既有取舍一致。

## 5. 实现顺序（分阶段，每阶段 verify）

- **S2.1 审计基座 + CRUD 织入**：`emit_rbac_write` helper + 5 域 service create/update/delete 加 `operator: AuditActor` 参数 + emit；api 层从 CurrentUser 构造传入。
- **S2.2 绑定 API**：`RbacBindingService`（deps 注入跨域 repository）4 写 + 4 读 + DTO + 绑定 router + 权限点 + 存在性/数据权限校验 + 绑定审计。
- 测试：绑定成功/空清空/去重/不存在 422/主体 404；权限 403 产 permission_denied；CRUD+绑定成功产 rbac_write、失败产 rbac_write failure、metadata 无敏感字段；非超管越权绑定被拦。

## 6. 待确认（已用 Codex 推荐默认实现，可调）

- `role-dept` 是否仅 `data_scope==custom_dept` 时允许写：**默认允许写**（非 custom_dept 也存，授权时不生效），不返 409；若需更严格再加 `role.DATA_SCOPE_NOT_CUSTOM`。
- 是否第一版即拆 `system:role:assign` 分权：**默认复用 edit**，产品明确分权需求再升级（属不可逆授权契约变化）。
- 成功响应 `200 BindingRead` vs `204`：**默认 200**（回显去重最终集合）。
