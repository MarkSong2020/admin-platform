## 一句话结论

> P1 RBAC 的灵魂不是「建几张表」，而是**声明式横切机制 + 默认 deny + 后端强校验 + 数据权限可测**（roadmap §9.2 戳破的盲点：RuoYi 真正该学的是 AOP 横切，不是表结构）。本 spec 把 4 个不可逆决策（超管 / data_scope 注入 / 缓存 / 前端契约）固化成机制，**先于任何建表**——否则 RBAC 沦为 CRUD 表集合，前端 P6 返工。

## 0. 本 spec 的角色（为什么先写它）

roadmap §4 工程红线 **P1.0**：先定权限装饰器/依赖、默认 deny、data_scope 注入策略、超管模型、权限缓存失效策略，并**冻结前端接口契约**，否则 RBAC 沦为 CRUD 表、前端返工。本 spec 即该红线的落地。

**边界**：
- ✅ 本 spec 定：横切机制（鉴权依赖 / data_scope helper / Provider 接口）、超管语义、前端契约形状、DoD 测试清单、分层落地路径、迁移规划。
- ❌ 本 spec 不定：各表完整字段 DDL（实现各模块时核对 RuoYi 源码）、dept 树具体存储结构（§11 开放问题）、缓存实现（P1 不做）。

## 2. 超管模型（Q6）

### 2.1 信任根 = 布尔（复用现状，不新造）

现状已落地，**不重复造**：
- `users.is_super_admin: Mapped[bool]`（`src/admin_platform/domains/user/models.py:38`）
- partial unique index `uq_users_one_super_admin`（`models.py:30-35`，`postgresql_where=text("is_super_admin")` → DB 层至多一行 true）
- 禁删最后超管：`service.py:71-78`（`count_super_admins() <= 1` → 抛 `admin_platform.LAST_SUPER_ADMIN`，409）
- 请求侧不接 `is_super_admin`：CLI bootstrap 专属（`schemas.py:4-7`）

### 2.2 至多一超管（保留约束，放开推迟 P2）

- **保留** `uq_users_one_super_admin` partial unique index 不动。
- 放开多超管是不可逆 schema 变更（`drop index`），P2 有了「受审计的超管增删用例 + 管理 UI + 二次确认」后再做。
- **理由**：当前只有 CLI bootstrap，无增删超管的受控用例；放开 = 打开爆炸半径却无管控（Codex 风险 2）。

### 2.3 超管短路（统一入口，覆盖授权、不绕安全）

- 短路点：**统一权限校验函数/依赖入口**（§3.2），**不放 middleware**。
- 短路**覆盖**：RBAC 权限点检查 + data_scope 5 范围（root 视作 `全部` 范围）。
- 短路**不绕过**：登录鉴权（`require_current_user`）、账号 `status` 停用检查（登录已有，见 P0.9）、审计落库（P2）、业务不变式（如禁删最后超管）、资源存在性（404 仍照常）。
- 落地：在权限依赖里 `if current_user.is_super_admin: return`（短路）放在所有 RBAC/scope 判断之前、但在 `require_current_user` 之后。

### 2.4 展示角色（解决前端漂移）

- 问题（Codex 风险 1）：纯布尔时超管的 `getInfo.roles` 为空 → 前端「有后端权限但菜单看不到自己」。
- 解法：给超管在 `getInfo`/`getRouters` 响应里**合成**一个展示角色（如 `roles: ["superadmin"]`）+ 全量菜单/权限，**但不依赖 DB 里真有这条角色记录**（权限真相仍是布尔，避免「超管角色被误删 = root 失效」）。
- DoD：测试断言超管的 `getInfo`/`getRouters` 返回全量可见（§9）。

## 3. 权限校验机制（默认 deny）

### 3.1 CurrentUser 扩展（权限不进 token）

现状 `CurrentUser`（`src/admin_platform/core/auth.py:80-87`）仅 `user_id / sub / scope`。P1 扩展需补：`is_super_admin: bool`、`roles: list[str]`、`permissions: list[str]`、`dept_id: int | None`、`data_scope`（见 §4）。

> ⚠️ **权限不塞 JWT claim**：`issue_access_token`（`security.py:86`）的 token 只带 `user_id / username`，**不带 roles/permissions**。权限由权限依赖在**每次请求时**经 `PermissionProvider` 从 DB 查（Q8 不缓存）填充 `CurrentUser`。
> **理由**：把权限塞 token 会让「改权限立即生效」失效（要等 token 刷新）——与 Q8「改权限立即生效」的安全预期冲突。这是 Q8「不缓存每次查 DB」的直接推论。

> ⚠️ **基础设施红线**：扩展 `CurrentUser` + 加权限依赖 = 碰 `core/auth.py`（AI_CODING_RULES §基础设施红线，`doc/standards/AI_CODING_RULES.md:103-120`）。加 dataclass 字段是低风险，但需谨慎；不改 `AuthMiddleware` 现有 token 解析与 `request.state` 注入语义。

### 3.2 权限依赖（默认 deny，后端强校验）

- 形态：FastAPI 依赖 `require_permission("system:user:list")`（返回一个 `Depends` 可调用），挂在路由上；超管短路（§2.3）在其内部最前。
- **默认 deny**：受保护资源**默认拒绝**，只有显式声明权限点 + 用户实际拥有该权限才放行。不靠前端隐藏按钮当安全（roadmap §4）。
- 错误码：复用已预留常量——`AUTH_FORBIDDEN_BY_ROLE = "auth.FORBIDDEN_BY_ROLE"`（`core/errors.py:104`）、`AUTH_FORBIDDEN_BY_SCOPE = "auth.FORBIDDEN_BY_SCOPE"`（`errors.py:105`），均 403。经 `AppError`（`errors.py:179-195`）抛出。
- 分层：权限依赖属顶层 api 基础设施（类似 `require_current_user`），不破坏 C2–C7（`.importlinter`）。

### 3.3 权限标识规范

- 权限点字符串对标若依 `perms`：`{模块}:{资源}:{动作}`（如 `system:role:add` / `system:user:list`）。
- 与本仓**错误码** `{service}.{ERROR_CODE}`（`NAMING_CONVENTIONS.md:20-42`）是两套命名，不混用：错误码是 `auth.FORBIDDEN_BY_ROLE`，权限点是 `system:user:list`。

## 4. 数据权限机制（Q7）

### 4.1 五范围（RuoYi 标准）

`全部` / `自定义部门` / `本部门` / `本部门及以下` / `仅本人`。范围来自用户所属角色的 `data_scope` 字段（多角色取并集，§11 待定合并语义）。

### 4.2 声明式 + typed 上下文 + 显式 helper（三段式）

> 核心取舍（Codex）：**声明式负责「谁需要数据权限」，显式 helper 负责「SQL 怎么加条件」**。不为了「像 RuoYi AOP」牺牲可测性。

1. **声明层**：api 用 `@data_scope(...)` 或依赖声明该端点受数据权限约束；依赖从 `CurrentUser` 解析出 typed `DataScope`（含范围枚举 + dept_id + 可见 dept 集合 + user_id），**不含 `Request`**。
2. **传递层**：service 接收 `DataScope` typed context（不传 `Request`/`Response`，守 C3）；透传给 repository。
3. **落地层**：repository 方法**显式**调用公共 `apply_data_scope(stmt, scope, *, dept_col, owner_col)`，在 `select` 上追加 `WHERE` 条件后再 execute。

### 4.3 不回 SQLAlchemy 事件（P0.9 教训）

- **禁止**用 `do_orm_execute` / `before_flush` 等事件做隐式全局查询改写——这正是 P0.9 拆 tenant_filter 的原因（隐式 / 全局 / 难测 / raw SQL 易绕过）。
- data_scope 必须是**显式、可在单测里直接断言**的查询构造。

### 4.4 漏调用防御（Codex 风险 3）

- 风险：repository 新增 list 查询忘记调 `apply_data_scope` → happy-path 测试照过、越权漏网。
- 缓解：①每个受数据权限约束的 list 端点**强制**权限矩阵负向测试覆盖 5 范围（§9）；②P1.0 评估一条约定/静态检查——带 `@data_scope` 的 operation 必须经过 scoped repository 方法（实现方式 P1 定，可先靠 review + 测试兜底）。

## 5. 权限 / 菜单 Provider（Q8）

### 5.1 P1 不缓存

- P1 **不做 Redis 权限缓存**，每次请求查 DB；至多单请求内 memoization（同一请求多次取权限不重复查）。
- 理由：P1 首要是正确性 + 矩阵测试 + 失效语义冻结，不是吞吐；「改权限立即生效」是后台管理平台安全预期。先把边界做对，比引入 Redis 再补一致性测试更稳。

### 5.2 Provider 接口（预留缓存口子）

定义抽象（实现 P1 走直查 DB 版）：
- `PermissionProvider`：`get_user_permissions(user_id) -> set[str]`、`get_effective_data_scope(user_id) -> DataScope`
- `MenuProvider`：`get_user_menu_tree(user_id) -> list[MenuNode]`
- 失效语义预留：`invalidate_user(user_id)`、`invalidate_role(role_id)`、`invalidate_all()`（P1 是 no-op，因无缓存；接口先冻结，P2 接 Redis 时填充）。

### 5.3 将来缓存方向（非 P1）

若 P2+ 加缓存：优先「角色/菜单变更**主动失效** + 权限**版本号**」，**不建议只靠短 TTL**（短 TTL 有撤权延迟窗口，权限场景不可接受）。

## 6. 前端接口契约冻结（Q9）

> 只冻结 RBAC payload 形状，为前端选 vben（薄适配）/ RuoYi-Vue3（零适配）留后路。**全局 API 风格不动**。

### 6.1 必冻字段（P1 接入前端即高返工，必须 P1 定）

- **getInfo**（当前登录用户信息）：
  - `user`：基础字段（id / username / nickname / 头像 / 状态…）
  - `roles`：角色 code 数组（超管合成 `["superadmin"]`，§2.4）
  - `permissions`：权限标识数组（超管返回 `["*:*:*"]` 或全量）
- **getRouters**（动态路由树）：
  - 核心字段 `name / path / component / redirect / hidden / alwaysShow / meta`
  - `meta.title / icon / noCache / link` 先固化

### 6.2 可后加字段（不阻塞 P1）

`breadcrumb / activeMenu / affix / cacheView / query / frameSrc` 等前端增强字段——后加不破坏已接入前端。

### 6.3 全局风格保留（不学若依）

- **错误响应**：保留本仓 RFC 9457 ProblemDetail（`core/errors.py` 全局 handler，AI_CODING_RULES 禁改 ProblemDetail 字段）——**不**改成若依 `{code,msg,data}`。
- **分页 envelope**：保留本仓 `{items,page,size,total,total_pages}`（`UserPage`，`schemas.py:47-55`）。
- **OpenAPI**：保留 `operation_id` / `tags` 规范。

### 6.4 端点路径（payload 兼容 + 路径本仓化）

- payload 按若依形状冻结；**路径走本仓 RESTful 风格**（如 `GET /api/v1/auth/user-info`、`GET /api/v1/menus/routers`），不强求字面 `/getInfo` `/getRouters`。
- 理由：本仓 OpenAPI/operation_id 规范不为路径破例；前端选 vben/RuoYi-Vue3 时配 API base + 薄适配层成本极低（roadmap §前端契约说明）。
- ⚠️ 若 P6 最终选「RuoYi-Vue3 零适配」，路径需再贴近若依——此点标记为 §11 开放项，**不阻塞 P1**（payload 冻结才是关键）。

## 7. 表结构方向（字段实现时核对 RuoYi 源码）

> 单租户下角色/菜单/部门/岗位**全部全局**（无 tenant_id）。具体字段 P1 各模块实现时核对 RuoYi-Vue3-FastAPI（roadmap §7 Q1），本节只定方向。

核心表（参考 roadmap §5）：
- `roles`（角色 + `data_scope` 数据权限范围字段）
- `menus`（菜单树：目录/菜单/按钮三类 + `perms` 权限标识 + 路由/组件元数据供前端动态路由）
- `depts`（部门树，数据权限载体）
- `posts`（岗位）
- 关联：`user_roles` · `role_menus` · `role_depts`（自定义数据权限）· `user_posts`

**适配本仓纪律**（DATA_MODELING.md + NAMING_CONVENTIONS.md）：
- `IdMixin`（BIGINT）+ `TimestampMixin`（`db/base.py:42-63`）
- 表名 = URL 复数（`roles` / `menus` / `depts` / `posts`），列 snake_case **必带中文 comment**（门禁 `tests/unit/test_column_comments.py`）
- 唯一约束注册到 `register_unique_constraint`（对标 `models.py:47-51`），错误码 `admin_platform.{X}_DUPLICATE`

## 8. 分层与生成器落地

- 新域**必走** `make new-module name=<域> with-model=1`（`Makefile:43-49`），生成五层骨架 + 自动 patch `migrations/env.py` 注册 model（`scripts/new_module.py:892-956`），**不手抄** user 域。
- ⚠️ **手工步骤（生成器不自动做，易漏）**：新域要手工加进 `.importlinter` 的 C1 `containers`（`.importlinter:23-31`，目前仅 user），否则新域不受 C1（api>service>repository 禁反向）机检。C2–C7 用 wildcard 自动覆盖。
- 迁移：从 `0003_one_super_admin`（当前 head）链接 `0004+`；`uv run alembic revision --autogenerate` 后人工 review（禁手改已发布迁移）。
- 分层硬约束（`LAYERED_DESIGN.md:93-99`）：api 不 import models/repository；service 不 import fastapi、只抛 AppError；repository 不抛 HTTPException；schemas 不混 ORM；models 不放序列化。

## 9. DoD — 权限矩阵负向测试清单（每模块强制）

> roadmap §4「每模块 DoD 红线」+ D8 关键路径门。每个 RBAC 模块合并前**必须**覆盖以下负向/边界（防 coverage 幻觉：total 达标但安全路径没测）：

- [ ] **无权限 403**：未授权用户访问受保护端点 → `auth.FORBIDDEN_BY_ROLE`
- [ ] **按钮权限 403**：有列表权限无删除权限的用户调删除 → 403
- [ ] **data_scope 五范围**：同一 list 端点在 `全部`/`自定义部门`/`本部门`/`本部门及以下`/`仅本人` 下结果集断言正确（含跨部门数据**不可见**的负向断言）
- [ ] **data_scope 漏调用**：scoped list repository 方法被断言确实调用了 `apply_data_scope`（§4.4）
- [ ] **超管短路**：超管 `getInfo`/`getRouters` 全量可见；超管跨部门 list 看到全部（短路 data_scope）；超管仍不能绕业务不变式（如禁删最后超管）
- [ ] **信任根保护**：最后一个 active 超管不能被禁用/删除（复用 `LAST_SUPER_ADMIN`）；请求侧改 `is_super_admin` 被拒
- [ ] **权限立即生效**：改角色权限/菜单后，同一用户**下一次请求**权限即变（验证 Q8 不缓存语义）
- [ ] 工程门：`pyright`（至少 auth/rbac/core 局部 strict，D8）+ `import-linter` 全绿 + 新域已入 C1 containers
- [ ] **契约机检（spec v2，§13）**：`test_permission_registry_contract`（registry 无重复 + 命名规范）+ `test_permission_menu_contract`（registry / 路由 / 菜单 perms 三组一致，`frontend_only` 豁免除外）+ `test_audit_event_contract`（三类事件产 `audit_event.v1` 且敏感字段不出现）+ `test_rbac_frontend_contract`（getInfo/getRouters schema + 示例 JSON 快照）

## 13. spec v2 增补：seed / 权限 registry / 审计 envelope / 契约产物（Q11/Q12/Q16/Q17）

> 2026-06-05 第二轮 Claude × Codex PK 收敛（四题双方独立一致，无实质分歧）。本章兼容 §1–§12 已冻结机制，不推翻。

### 13.1 RBAC 初始数据 seed（Q11）

**决策**：菜单/权限/内置角色用**版本化 seed manifest**（Python typed data，便于复用 §13.2 registry 常量 + 类型检查）作真相源，由幂等 `rbac seed` CLI 命令 upsert；超管 `create-super-admin` CLI **保持独立**（信任根特殊，不并入普通 seed 重跑流程）。

- **不进 Alembic data migration**：Alembic 只管 schema migration + drift（`make check-db` = `alembic check`）；业务/参考数据进迁移会让漂移门禁与初始化职责混淆。
- **幂等 + 防覆盖**：manifest 只管理 `system_managed=true` 的内置项；用户自定义菜单/角色**禁止 seed 覆盖**（Codex 风险 1）。
- **菜单版本化**：菜单增删改 manifest 文件（git 版本化），重跑 `rbac seed` 幂等 apply；不跟 Alembic 版本链。
- **测试复用**：pytest fixture 复用同一 seed loader（不另写一套测试 seed）。
- **展示角色**：seed 一条 `superadmin` 展示角色，**注明「不作信任根」**（信任根仍是 §2.1 布尔；解决 §2.4 前端漂移 + 收敛 §11 O3）。

### 13.2 权限点命名 + registry（Q12）

**决策**：**代码侧集中 `PermissionRegistry`（或常量模块）是权限真相源**；装饰器引用常量 `require_permission(Permissions.SYSTEM_USER_LIST)`，不写裸字符串；三组集合机检保持一致。

- **❌ 反模式（明确禁）**：让 DB `menus.perms` 反推代码权限真相。默认 deny 的本质是**端点声明保护**，DB 菜单只承载前端展示/角色授权；菜单漏挂 perms 不能等于后端失保护（Codex 裁决信号的反对路径）。
- **三组集合双向一致**：① registry 声明的权限点；② 路由 `@require_permission` 实际使用集；③ seed 菜单 `menus.perms`。机检断言三者一致（无悬空权限点、无失保护端点、无菜单挂不存在的 perms）。
- **显式豁免**：纯菜单目录（无 perms）、纯前端按钮（有展示无后端端点）标 `frontend_only`，避免双向机检误伤（Codex 风险 2）；豁免必须显式声明，不放松全局检查。
- **机检**（§9 DoD 补）：`test_permission_registry_contract`（无重复 + 命名符合 `system:{resource}:{action}`）、`test_permission_menu_contract`（三组集合一致，豁免除外）。

### 13.3 审计事件 envelope（Q16，P1 冻字段契约 / P2 织入实现）

**决策**：P1 冻结 `audit_event.v1` **内部事件模型**（带 `schema_version`）；P1 只在三类事件落**最小 hook**（先结构化日志），P2 做完整中间件 + 持久化表，**保持字段兼容**。

- **字段（冻结 v1）**：`schema_version` · `event_id` · `event_type` · `action` · `title` · `occurred_at_utc` · `actor{user_id, username, is_super_admin}` · `target{type, id, display}` · `request{request_id, trace_id, method, path, ip, user_agent}` · `result{status, http_status, error_code}` · `duration_ms` · `risk_level` · `metadata`（扩展位）· `redaction_applied`。
- **P1 必落三类事件**：权限拒绝（403）/ 登录失败 / RBAC 写操作（超管不绕审计，呼应 §2.3）。
- **与 ProblemDetail 关系**：**分离**。审计是内部事件 ≠ 对外错误响应（§6.3 保留 RFC 9457）；二者只共享 `request_id / trace_id / error_code` 作关联键，不互相塞字段。
- **脱敏双层**（Codex 风险 3）：deny-list（`Authorization` / `password` / `token` 永不入审计）+ allow-list（只记录白名单字段）；`redaction_applied` 标记是否脱敏过。
- **向后兼容**：`schema_version` + `metadata` 扩展位保证 P2 迁表/扩字段不破坏 v1 契约。
- **机检**（§9 DoD 补）：`test_audit_event_contract`（三类事件都产 `audit_event.v1` 且敏感字段不出现）。

### 13.4 前端契约产物形态（Q17，P1 冻结可机检）

**决策**：**OpenAPI schema（类型）+ 示例 JSON（若依 payload 语义）+ contract test（机检冻结）**；P1 **不做 mock server**（留 P6 / 前端并行开发时再做）。

- **三者分工**：OpenAPI 由 response_model 自然产出管类型；示例 JSON 承载 OpenAPI 校验不了的若依语义（`meta.title/icon/noCache/link`、超管 `roles/permissions`）；contract test 抓 payload 漂移。
- **复用现有先例**：扩展 `tests/unit/test_openapi_contract.py` 的「规则表 + 违规探针」模式做 RBAC 子集契约测试（getInfo/getRouters response schema 必含字段 + 示例能过 Pydantic model + 破坏字段触发探针）。
- **样本路径**：契约样本放 `tests/contracts/`（测试可读优先），文档引用它，避免双份漂移。
- **机检**（§9 DoD 补）：`test_rbac_frontend_contract`（getInfo/getRouters schema + 示例 JSON 快照）。
