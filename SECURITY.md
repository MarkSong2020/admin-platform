# 安全策略

## 支持的版本

本项目处于活跃开发期，安全修复只针对默认分支 `main` 的最新代码。

| 版本 | 是否接受安全报告 |
|---|---|
| `main`（最新） | ✅ |
| 历史 tag / 派生 Fork | ❌（请自行同步上游修复） |

## 报告漏洞

**请勿在公开 issue 中披露安全漏洞。**

请通过 GitHub 仓库 **Security → Advisories → Report a vulnerability** 提交私密报告。报告时请尽量包含：

- 受影响的端点 / 模块 / 文件路径
- 复现步骤或 PoC
- 影响评估（数据泄露 / 越权 / RCE / 拒绝服务等）
- 你认为合理的修复方向（可选）

## 响应流程

- 我们会在收到报告后尽快确认，并就严重程度与修复计划与你沟通。
- 修复发布前，请给予合理的协调披露窗口（coordinated disclosure），不要提前公开细节。
- 修复合入后，会在 `CHANGELOG.md` 记录（默认致谢报告者，除非你要求匿名）。

## 范围

- **范围内**：本仓库自身的应用代码（`src/`）、迁移、配置默认值、依赖配置。
- **范围外**：第三方依赖自身的漏洞（请报告给对应上游，本项目会跟进升级）；使用者在派生 Fork 中引入的改动；明显属于错误部署配置（如生产开启 debug、`*` CORS）而非代码缺陷的问题。

## 已内置的安全基线

本项目在框架层落地了多项 defense-in-depth 实践（输入校验、参数化查询、文件上传扩展名白名单 + 魔数校验、Excel formula injection 防御、`Content-Disposition` 注入防御、定时任务 handler 白名单防 RCE、错误响应生产脱敏等）。完整清单与正本见 [`docs/STANDARDS.md`](./docs/STANDARDS.md) 的「安全基线」段。
